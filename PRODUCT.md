# Product

## Register

product

## Users

Developers and design-system authors evaluating Control UI components, skins, registry manifests, and runner integration examples inside the docs app.

## Product Purpose

Control UI is a shadcn-registry-first source system for production AI interfaces. The docs app should make the installable source, token model, skins, and runtime-agnostic component contracts easy to inspect, try, copy, and adapt.

## Brand Personality

Calm, precise, and pragmatic. The interface should feel like a serious developer tool rather than a marketing gallery.

## Anti-references

Avoid decorative SaaS dashboards, card-heavy component galleries, oversized marketing copy, noisy theme controls, and runtime-provider coupling that hides the source model.

## Design Principles

- Registry source is the product; demos and docs should read from real files.
- Show the token and skin model directly instead of hiding it behind vague presets.
- Keep operational surfaces dense, scannable, and quiet.
- Prefer CSS-first behavior and theme tokens over local JavaScript state or one-off styles.
- Use familiar shadcn-compatible control vocabulary so the docs feel ownable after install.

## Accessibility & Inclusion

Prioritize keyboard access, readable contrast, reduced-motion support, visible focus states, and controls whose labels expose the affected CSS variables.
